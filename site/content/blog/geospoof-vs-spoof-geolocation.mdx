Changing the location a website sees is the easy part. Any extension can set
`navigator.geolocation` to coordinates in Tokyo. The hard part is making every
*other* signal agree, because a site that catches one disagreement doesn't see
Tokyo — it sees someone pretending to be in Tokyo, which is worse than not
hiding at all.

So instead of comparing feature lists, we put Spoof Geolocation and
[GeoSpoof](/) through the same test: our own
[browser location test page](/verify), which probes the full stack of signals a
real fingerprinting script would check. Same browser, same target location, same
checks. Every screenshot below is a live result.

[Spoof Geolocation](https://chromewebstore.google.com/detail/spoof-geolocation/ihdobppgelceaoeojmhpmbnaljhhmhlc) is the natural
benchmark here: with over 100,000 users on the Chrome Web Store, it's the most
widely used location spoofer on both Chrome and Firefox. If you've searched for
a way to change your browser's location, it's probably the first extension you
found — which is exactly why it's worth holding to a full fingerprint test
rather than taking "it moves the map pin" at face value.

## How the test works

The [verify page](/verify) doesn't just read back the location you set. It pulls
each signal independently — the timezone offset, individual `Date` methods,
locale formatting, and a set of exotic execution contexts — then checks whether
they tell the same story. A green result means the signals are internally
consistent. A red one means a site could spot the contradiction.

If you want to follow along, install either extension, set a location, and open
the test yourself. Nothing here is hand-picked; it's just what the page returns.

## 4. Timezone and offsets

Setting the timezone *name* is easy. The trap is the **offset**. The UTC offset
for a zone has to match the location the pin is sitting in, and it has to be
right for *any* date, since daylight-saving rules have changed over the years.

This is where Spoof Geolocation comes apart. It fails every non-UTC timezone
check — the offset it reports is UTC's, not the spoofed location's. So the map
says one place and the clock says Greenwich. GeoSpoof computes the offset from
the zone's full history, so it stays consistent no matter where the pin is or
which date a script picks.

## Why this matters

Spoof Geolocation's failures all trace back to one root cause: it moves the
coordinates but never shifts the clock off UTC. That single gap cascades through
every time-based signal — the offset, the `Date` methods, the string methods, and
`Intl` all inherit it. The result is a map pin in one place and a clock in
Greenwich, which is the exact contradiction
[a site cross-checks for](/blog/how-websites-know-your-location). Unless you
happen to be spoofing to a location that's actually on UTC, the disagreement is
there for any script to read.

That's the difference the test surfaces: Spoof Geolocation changes the location,
but the supporting signals stay on UTC. GeoSpoof's whole design goal is that they
follow the pin, so a site checking any of them sees one coherent location instead
of a seam. If you're shopping for a Spoof Geolocation alternative, that
consistency — not the length of a feature list — is what separates the best
location spoofing extension from one that just moves the map pin.
